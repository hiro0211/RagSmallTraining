# RAGの技術概要

RAG（Retrieval-Augmented Generation）とは、検索拡張生成と呼ばれる技術です。大規模言語モデル（LLM）の回答精度を向上させるために、外部のナレッジベースから関連情報を検索し、その情報をコンテキストとしてLLMに提供する手法です。

RAGの基本的な処理フローは以下の通りです：
1. ユーザーが質問を入力する
2. 質問文をベクトル化（Embedding）する
3. ベクトルデータベースで類似度検索を行い、関連ドキュメントを取得する
4. 取得したドキュメントをコンテキストとしてLLMのプロンプトに組み込む
5. LLMがコンテキストに基づいて回答を生成する

RAGの利点は、LLMの学習データに含まれない最新情報や社内固有の知識に基づいた回答が可能になることです。また、回答の根拠となるソースドキュメントを提示できるため、回答の信頼性を高めることができます。

# ベクトル検索の仕組み

ベクトル検索（Vector Search）は、テキストや画像などのデータを数値ベクトル（Embedding）に変換し、ベクトル間の距離や類似度を計算して関連するデータを検索する技術です。

テキストのEmbeddingには、OpenAIのtext-embedding-3-smallなどのモデルが使用されます。このモデルはテキストを1536次元のベクトルに変換します。例えば「犬」と「猫」のベクトルは「犬」と「自動車」のベクトルよりも距離が近くなります。

類似度の計算にはコサイン類似度が一般的に使用されます。コサイン類似度は-1から1の値を取り、1に近いほど類似していることを示します。ベクトル検索では、クエリベクトルとデータベース内の全ベクトルとの類似度を計算し、最も類似度の高いものから順に結果を返します。

pgvectorは、PostgreSQLにベクトルデータ型とベクトル検索機能を追加する拡張です。HNSWインデックスを使用することで、高速な近似最近傍探索が可能になります。

# LangChainの基本概念

LangChainは、大規模言語モデル（LLM）を活用したアプリケーション開発のためのフレームワークです。Python版とJavaScript/TypeScript版が提供されています。

LangChainの主要なコンポーネントは以下の通りです：
- **Models**: OpenAI、Anthropicなど各社のLLM APIへの統一的なインターフェース
- **Prompts**: プロンプトテンプレートの管理と動的なプロンプト生成
- **Chains**: 複数のコンポーネントを連結して処理パイプラインを構築
- **Memory**: 会話履歴の管理と文脈の保持
- **Retrieval**: ドキュメントの読み込み、チャンク分割、ベクトル検索

LangChainを使用することで、RAGシステムの構築が大幅に簡略化されます。ドキュメントローダーでPDFやテキストファイルを読み込み、テキストスプリッターでチャンク分割し、Embeddingモデルでベクトル化し、ベクトルストアに格納するという一連の処理を、少ないコードで実装できます。

# ハイブリッド検索

ハイブリッド検索は、ベクトル検索（セマンティック検索）とキーワード検索（BM25/全文検索）を組み合わせた検索手法です。それぞれの弱点を補完し合うことで、単独の検索方式よりも高い精度を実現します。

ベクトル検索は意味的に近い文書を見つけることが得意ですが、固有名詞や専門用語の完全一致を保証しません。一方、キーワード検索は文字列の一致に強いですが、言い換えや類義語を見逃します。ハイブリッド検索では、両方のスコアを加重平均して最終的な類似度スコアを算出します。

一般的な重み配分はベクトル検索70%、キーワード検索30%です。この比率はドメインやユースケースに応じて調整します。例えば、法律文書のように正確な用語一致が重要な場合はキーワード検索の比率を上げ、カジュアルな質問が多い場合はベクトル検索の比率を上げます。

PostgreSQLでのハイブリッド検索は、pgvectorのベクトル類似度検索とPostgreSQLの全文検索（tsvector/tsquery）を組み合わせて実装します。GINインデックスを使用することで、全文検索のパフォーマンスを向上させることができます。

# チャンキング戦略

チャンキングとは、長いドキュメントを検索に適した小さな単位（チャンク）に分割する処理です。チャンクの品質はRAGシステムの検索精度に直結するため、適切な戦略の選択が重要です。

## 固定長チャンキング

最もシンプルな方法で、テキストを一定の文字数（例: 500文字）ごとに分割します。オーバーラップ（重複部分）を設けることで、分割境界での文脈の断絶を軽減します。実装が簡単ですが、文の途中で切れたり、意味的に無関係な内容が1つのチャンクに混在する可能性があります。

## Markdown見出し分割

Markdownドキュメントの場合、見出し（h1, h2, h3）を区切りとして分割します。見出しは通常トピックの境界を示すため、意味的に一貫したチャンクが得られやすいです。ただし、見出し間のテキストが長い場合は、さらにサイズベースの分割を組み合わせます。

## セマンティックチャンキング

文間の意味的類似度を計算し、類似度が大きく低下する箇所で分割する手法です。各文をEmbeddingに変換し、隣接する文同士のコサイン類似度を計算します。類似度が閾値以下になった箇所がトピックの境界と判断され、そこで分割されます。

セマンティックチャンキングの利点は、内容の一貫性が保たれたチャンクが生成されることです。固定長分割では「ベクトル検索の説明の途中」で切れることがありますが、セマンティックチャンキングでは「ベクトル検索の説明」と「次のトピック」の境界で自然に分割されます。

閾値の設定方法には、固定値（例: 0.5以下で分割）、パーセンタイル（類似度が低い上位N%で分割）、標準偏差（平均から一定の偏差で分割）があります。パーセンタイル方式はドキュメントの内容に応じて自動調整されるため、汎用性が高いです。

# Reranking（再ランキング）

Rerankingは、初回検索で取得した候補ドキュメントを、より精密なモデルで再スコアリングして順位を並べ替える手法です。初回検索では高速だが粗い検索（Bi-Encoder）を行い、Rerankingでは低速だが精密な検索（Cross-Encoder）で上位を絞り込みます。

## Bi-Encoder vs Cross-Encoder

Bi-Encoderは、クエリとドキュメントをそれぞれ独立にベクトル化し、ベクトル間の距離で類似度を計算します。事前にドキュメントをベクトル化しておけるため高速ですが、クエリとドキュメントの相互作用を考慮できません。

Cross-Encoderは、クエリとドキュメントのペアを同時に入力し、直接的に関連度スコアを出力します。両者の相互作用を考慮できるため精度が高いですが、全ドキュメントとのペアを計算する必要があるため低速です。

## 2段階検索パイプライン

実用的なRAGシステムでは、Bi-Encoder（ベクトル検索）で広く候補を取得し（例: 10〜20件）、Cross-Encoderで精密に再スコアリングして上位（例: 5件）に絞る2段階パイプラインが一般的です。これにより、速度と精度のバランスを取ることができます。

代表的なCross-Encoderモデルとして、cross-encoder/ms-marco-MiniLM-L-6-v2があります。このモデルは軽量で、CPU環境でも実用的な速度で動作します。

# クエリ変換

クエリ変換は、ユーザーの質問をそのまま検索に使うのではなく、検索に適した形に変換する技術群です。ユーザーの質問は曖昧だったり、検索に最適な表現ではないことが多いため、変換によって検索精度を向上させます。

## クエリリライト

最もシンプルなクエリ変換で、略語の展開（RAG → Retrieval-Augmented Generation）、タイポの修正（べくとる → ベクトル）、曖昧な質問の明確化（ベクトルか → ベクトル検索とは何ですか）を行います。LLMを使って自動的にリライトします。

## HyDE（Hypothetical Document Embeddings）

HyDEは、ユーザーの質問に対してLLMが仮説的な回答を生成し、その回答のEmbeddingで検索を行う手法です。ユーザーの短い質問よりも、LLMが生成した詳細な仮説回答のほうが、実際のドキュメントに近いEmbeddingになるという考えに基づいています。

例えば「ベクトル検索とは？」という短い質問のEmbeddingよりも、LLMが生成した「ベクトル検索は、テキストを数値ベクトルに変換し、コサイン類似度で関連ドキュメントを検索する技術です。OpenAIのtext-embedding-3-smallなどのモデルを使用して...」という仮説回答のEmbeddingのほうが、実際のドキュメントに近い位置にマッピングされます。

## Multi-Query Expansion

Multi-Query Expansionは、1つの質問をLLMで3〜5の異なる言い換えに展開し、それぞれで並列に検索を行い、結果を統合する手法です。表現の違いによる検索漏れを防ぎ、Recall（検索の網羅性）を向上させます。

例えば「ベクトル検索とは？」という質問から以下のような言い換えを生成します：
- 「ベクトル検索の仕組みを教えてください」
- 「Embedding検索とは何ですか」
- 「ベクトル類似度検索の技術的な説明」
- 「セマンティック検索の原理」

各クエリで検索した結果を統合し、重複を除去してからRerankingに渡します。

# RAG評価フレームワーク（RAGAS）

RAGASは、RAGシステムの品質を定量的に評価するためのフレームワークです。以下の4つの主要指標を計測します。

## Context Recall（コンテキスト再現率）

検索結果に、正解回答を導くために必要な情報がどれだけ含まれているかを測定します。正解回答（Ground Truth）の各文が、検索されたコンテキストのいずれかに帰属できるかを判定します。スコアが低い場合、検索が必要な情報を取りこぼしていることを示します。

## Context Precision（コンテキスト精度）

検索結果の上位に、正解に関連する情報がどれだけ集中しているかを測定します。関連するチャンクが上位にあるほどスコアが高くなります。Rerankingの効果を測定するのに適した指標です。

## Faithfulness（忠実度）

LLMの回答が、検索されたコンテキストの情報に忠実であるかを測定します。回答に含まれる各主張（claim）が、コンテキストから裏付けられるかを判定します。スコアが低い場合、LLMがコンテキストにない情報を捏造（ハルシネーション）していることを示します。

## Answer Relevancy（回答関連度）

LLMの回答が、ユーザーの質問にどれだけ関連しているかを測定します。回答から逆に質問を生成し、元の質問との類似度を計算します。スコアが低い場合、回答が質問の意図からずれていることを示します。

これら4指標を組み合わせることで、RAGシステムの「検索の品質」と「生成の品質」の両面を定量的に評価できます。改善施策の前後で指標を比較することで、各施策の効果を客観的に判断できます。

# メタデータフィルタリング

メタデータフィルタリングは、ベクトル検索の前段階でメタデータ条件に基づいてドキュメントを絞り込む手法です。検索対象を事前に限定することで、不要なチャンクがヒットすることを防ぎ、検索精度と速度の両方を向上させます。

一般的なメタデータ項目には以下があります：
- ドキュメント種別（source_type）: メソッド文書、CRMデータ、チャット記録など
- 作成日（created_at）: 鮮度に基づくフィルタリングやブースト
- カテゴリ（category）: 営業手法、顧客開拓、商談管理など
- 重要度（importance）: 重要なドキュメントを優先的に検索
- 言語（language）: 多言語環境での言語フィルタ

PostgreSQLのJSONB型でメタデータを格納し、GINインデックスを作成することで、メタデータ条件による高速なフィルタリングが可能になります。検索クエリにメタデータ条件を追加することで、例えば「営業手法カテゴリのドキュメントのみから検索」といった絞り込みが実現できます。
